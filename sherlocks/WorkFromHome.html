Task 4

The threat actor discovered a privilege that allows specific volume-level management operations and could be exploited to get full control over the C drive. What is this special privilege?

I know exactly what this privilege is, it is SeManageVolumePrivilege. This allows a user to perform volume-level management operations, which can be abused (via tools like SeManageVolumeExploit) to reset the DACL permissions on the entire C: volume, effectively giving every user access to otherwise-protected files like NTDS.dit or a CA's private key.

These two fields tell slightly different parts of the story. The Dangerous Host danger type confirms that Chrome's safe browsing flagged the download source as malicious. The Browser Shutdown interrupt reason tells us how the download actually ended: the browser was closed.

Most likely, the attacker saw the safe browsing warning popup and closed the browser, either out of frustration or to avoid leaving more traces, before pivoting to a quieter alternative.

Task 8

Since the download was not successful from the browser directly, which LOLBIN did the threat actor use to download this file successfully?

A few LOLBINs come to mind for a Windows machine, curl and certutil being the first ones. I hoped to find out more by looking through their prefetch files, but the prefetch directory didn't contain any mentions of either.

I used MFTECmd.exe to parse the $J and $MFT journals, and loaded the latter into Timeline Explorer. I then set up these filters:

[Update Timestamp] > #2025-05-28 12:44:01# And Contains([Extension], '.dll') And Contains([Name], 'Print')

I've seen this [1] file name convention before. It's a telltale sign of certutil usage.

The reason this [1] suffix exists is that certutil initially downloads the file into the user's temporary internet files cache at C:\Users\otello.j\AppData\LocalLow\Microsoft\CryptnetUrlCache\Content. To prevent overwriting files with the same name, the Windows Cache Manager appends a number in brackets to the new download. Once the download is complete in the cache, certutil copies/moves the data from there to the final destination specified.

Here we can see exactly how and why Defender detected this DLL as malicious and quarantined it.

RTP activated roughly around 15:19:19 and began querying the found DLL shortly after. After getting a few low-confidence matches from its internal signature database, it issued an SDN query, reaching out to Microsoft's cloud with the malware's hash to check whether other machines had globally reported it as malicious.

After the query completed, the DLL was identified as Trojan:Win64/Meterpreter.E, and Defender defaulted to quarantining it, flagging the PrintNotify service in the process as well, to prevent it from executing malicious code in memory.

Why did Defender wait almost 3 hours before killing the malware? My best guess is that a change was detected in a critical directory (but then why not right away?), OR the PrintNotify service tried to actually use the DLL, which prompted RTP to kick in and nuke it.

To answer the question, the malware was detected as a Meterpreter payload at 2025-05-28T15:19:35.

PrintNotify's AppID is {588E10FA-0618-48A1-BE2F-0AD93E899FCC}. An AppID is used by Windows to associate a COM server (like a service) with its configuration and security settings, and is tied to a CLSID that identifies the actual COM class being instantiated.

No tzres.dll, which confirms that systeminfo does not load this library directly. The next question then is: what does load DLLs from the wbem directory during a systeminfo run?

systeminfo.exe uses COM to interact with WMI, typically via calls like CoCreateInstance. This results in WMI client-side components such as wbemprox.dll being loaded into its process.

The request is then handled by the WMI infrastructure (Winmgmt), which coordinates execution out-of-process. Instead of running providers in-process, WMI uses WmiPrvSE.exe (located in C:\Windows\System32\Wbem\) as a dedicated provider host. Critically, WmiPrvSE.exe loads its provider DLLs from the wbem directory, the exact same location where the attacker planted the malicious tzres.dll.

So when systeminfo triggers WMI, WmiPrvSE.exe spins up, loads provider DLLs including any present in the wbem folder, and the hijacked tzres.dll gets loaded alongside the legitimate ones.

WmiPrvSE.exe loads the necessary provider DLLs (such as CIMWin32.dll) to gather system information like CPU, RAM, hotfixes, and time zone data. Once collected, the data is returned to the client through COM marshaling (over RPC internally), where wbemprox.dll delivers it back to systeminfo.exe.
